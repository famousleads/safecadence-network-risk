# v9.37 ŌĆö Compliance section audit

Same audit-then-fix exercise as v9.33 / v9.35 / v9.36. The Compliance
section covers seven surfaces; **all seven are real, wired, and have
no trust violations**. v9.27 + v9.31 + v9.32 specifically polished
this surface across many cycles, and the work shows.

This release is therefore narrow: an audit doc, one minor UI gap
(brownfield import shortcut), and a small UX improvement on the
findings flow.

## Strong trust posture

- **Every commit shows a diff first.** `/findings` auto-fix renders the
  per-system diff card with `renderDiffCard()` *before* `commitFromDiff()`
  fires, and the commit requires the HMAC-bound confirm-token (TTL 600s,
  bound to IR hash + scope + actor + adapter version). v9.33 fixed this.
- **Policy preview is honest about its scope.** `/policies/new` calls
  `POST /api/policy/preview-config` which renders shape-only previews ŌĆö
  the real translators run at enforce-time. The UI says so plainly.
- **Default mode is `report_only`.** `quick_save_policy` defaults the
  YAML's `mode` field to `report_only` so a freshly-authored policy
  never silently goes live without an explicit operator flip.
- **Evidence packs are tamper-evident.** v9.27 #9 added a hash chain
  to the evidence store; v9.31 #2 wired the daemon to auto-append.
  Verified ŌĆö the API returns generation status and the chain head.
- **Drift reads three sources.** Policy drift (regressions: passŌåÆfail
  per policy), baseline drift (config baselines per asset), and
  cross-system drift (15+ detectors per v6.2.1 #4) all surface in
  `GET /api/drift/all` with a unified summary.
- **Risks have full CRUD with validation.** `/api/compliance/risks`
  GET/POST/DELETE accept structured input (likelihood, impact, owner,
  mitigation) and the endpoint validates required keys.

## Findings

### #1 ŌĆö `/policies/new` doesn't surface brownfield import

The endpoints `POST /api/policy/import-from-config` and
`POST /api/policy/import-from-fleet` exist (added in v9.32 #1) and
work ŌĆö they parse an existing config blob and infer policy IR. But
the YAML editor at `/policies/new` only exposes "Load Template" and
"Save". An operator who wants to start from a real config has to know
the API exists.

**Severity:** UX gap, not a trust violation. Ship as a small UI add
in v9.37: a "Load from running config" button next to "Load Template"
that opens a slide-over with a textarea + asset selector and POSTs to
the import endpoint.

### #2 ŌĆö `/findings` auto-fix has no bulk action

Each finding has its own auto-fix flow. Operators with 30+ findings
clicking through one-at-a-time is friction.

**Severity:** UX gap. Park for v9.38 ŌĆö bulk operations should land
across `/findings`, `/risks`, and `/queue` together.

### #3 ŌĆö `/evidence` pack generation has no progress / retry

Generating a SOC2 pack against a large fleet takes 10-30s. The UI
shows "generating..." but if the browser default fetch timeout
(~30s) trips, the user sees `error` with no resume path.

**Severity:** UX gap. Park for v9.38 ŌĆö needs an async job pattern
(202 Accepted + GET /status/{job_id}) which is a bigger lift than
fits a polish release.

### #4 ŌĆö `/policies` action buttons "(coming v9.2)" labels still present

The slide-over in `_POLICIES_SCRIPT` shows "Preview per-vendor" and
"Add exception" with `(coming v9.2)` next to them. Both features
shipped (v9.31 + v9.32). The labels are stale.

**Severity:** Cosmetic + slightly false advertising of incompleteness.
Fix in v9.37 ŌĆö remove the `(coming v9.2)` parenthetical and wire the
buttons to the existing endpoints (`/api/policy/preview-config` and
`/api/policy/changes`).

### #5 ŌĆö Confirm-token in URL parameter

The `/findings` apply path puts the HMAC-bound confirm-token in a URL
query string. Token is signed (no forgery), bound to the actor (no
cross-user replay), and 600s TTL ŌĆö so this is **acceptable per the
v9.33 design**, not a finding to fix. Documenting it here so the
auditor doesn't re-flag it.
