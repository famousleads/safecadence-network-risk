# v9.35 — Execute section audit

Same exercise we ran for identity in v9.33: walk every page +
endpoint + engine module that backs the Execute section, separate
what's actually working from what's surface-level, and use the
result to scope the next release.

Engine modules under `src/safecadence/execution/`:

* `schema.py` — `JobStatus`, `RiskLevel`, `CommandMode`, `RollbackPlan`,
  `ApprovalRequest`. Dataclasses only.
* `store.py` — JSON-backed file store (air-gap safe). Append-only audit log.
* `rbac.py` — 6-tier matrix; authors can never approve own jobs;
  `EXECUTE_REAL` not granted by default to any tier.
* `guardrails.py` — risk classifier (31 heuristics), blocked-command
  list, lockout detection (`no aaa new-model`, `no transport input ssh`,
  …). Unknown defaults to MEDIUM (safe).
* `workflow.py` — state machine
  (`DRAFT → REVIEW → APPROVED → DONE/FAILED`). Multi-approver for
  CRITICAL.
* `executor.py` — dry-run + real execution. Vendor targeting.
  Exporter: Ansible / Salt / NSO / raw / Markdown.
* `builder.py` — NL intent → per-vendor commands. 30 offline packs.
* `tier3.py` — SSH executor. Triple-gated:
  `SC_TIER3_ENABLED=1` env + `EXECUTE_REAL` capability + `acknowledge`
  + `i_mean_it` flags. TOTP MFA per-job challenge.

## B. Trust posture findings

### Strong (don't rebuild)

1. RBAC: 6 tiers, no role has `EXECUTE_REAL` by default. Authors
   cannot approve their own jobs (workflow.py:147).
2. Multi-approver gate for CRITICAL risk (rbac.py:57).
3. Dry-run path creates a real `CommandExecution` row with
   `dry_run=True` so the audit trail is authentic
   (executor.py:122).
4. Blocked-commands list rejects obviously destructive patterns
   (rm -rf, format flash, etc.).
5. Lockout detection escalates risk to CRITICAL when a job would
   sever the management plane.
6. Emergency stop: a flag file at `~/.safecadence/EMERGENCY_STOP`
   halts the executor between commands (tier3.py:79).
7. Tier-3 triple-gate is correctly paranoid. SSH cannot run unless
   the operator explicitly turned it on AND has the role AND
   acknowledged. TOTP MFA per-job-challenge required.

### Concerns

1. **Approval notification gap** — entering `REVIEW` does not page
   the approver. They have to poll `/approvals` (workflow.py:122
   creates the record but no notifier fires).
2. **Rate-limiting flag exists, isn't enforced.** `Job.rate_limit_per_minute`
   in schema; the executor loop ignores it.
3. **Builder docstring promises AI fallback that doesn't exist.**
   `builder.py:20-21` says AI mode kicks in when offline packs
   miss; the code path doesn't.

## C. Half-baked items (the v9.35 work list)

### #1 Rollback plan generator

`schema.py:170` defines `RollbackPlan(asset_rollbacks: dict[str, list[str]])`.
`workflow.py:160` calls `_generate_rollback_plan(job)` which returns
an EMPTY plan and stores it. So clicking Rollback marks the job
`ROLLED_BACK` in the audit log but produces zero undo commands.

Same pattern as the identity `_rollback()` lie I caught in v9.33 #3:
the data structure exists, the audit log says the right things, but
the actual operational behavior is a no-op. Trust regression.

**Fix in v9.35 #2:** invert each pack's commands. Instantiate the
plan with the per-device, per-command undo list. `/rollback` page
renders it. Tests pin the inversion for each of the 30 packs.

### #2 `/per-device-diff` is a table, not a diff

`_PDD_BODY` in v9_pages.py renders the executions table — useful
for "did command X run on device Y?" but not for "what changed in
the running-config?" The v9.22 ship description ("Real
/per-device-diff page") was technically true but operationally
shallow.

**Fix in v9.35 #3:** capture pre + post `show running-config` per
asset, render unified-diff, cross-link from /queue and the asset
cockpit.

### #3 Approval notifications

`workflow.request_approval()` writes the `ApprovalRequest` record
and the audit log entry, but doesn't wire to Slack/Teams/PagerDuty.
The notifier infrastructure exists (`safecadence.notifiers.*`) and
fires on findings; it's just not called from the workflow.

**Fix in v9.35 #4:** structured payload to the configured notifier.
Best-effort.

### #4 Builder AI fallback (advertised, not wired)

`builder.py:20-21` docstring:

> If intent doesn't match any keyword, [the builder] falls back to
> the configured AI provider for plain-English-to-commands
> translation. Falls back further to manual mode if AI is disabled.

The actual code: returns `[]` when the intent doesn't match a
pack. No AI call. No fallback chain.

**Fix in v9.35 #5:** call the existing `safecadence.ai.client`
provider when packs miss. Honor `SC_AI_DISABLED`. Document the
fallback chain so the docstring matches reality.

### #5 Tier-3 SSH output capture is skeletal

`tier3.py:144` opens the paramiko session, `_run_command()` sends
the command, but the read-back loop just collects bytes until a
prompt-like pattern. No structured parsing, no per-command
attached output, no error detection beyond TCP failure. The
`CommandExecution.outputs` list ends up with one summary blob.

**Fix in v9.35 #6:** read until prompt with per-command timeout,
attach a `CommandOutput` row per command with stdout/return-code,
strip vendor banners, detect known error patterns (`% Invalid
input`, `Error:`).

## D. Test coverage gaps

Existing coverage (`tests/policy/test_v7_0.py`, `test_v7_1.py`):

* RBAC matrix shape, no-role-has-execute-real-by-default
* Guardrails classifier, blocked-commands, lockout heuristics
* Tier-3 triple-gate refusal cases
* Workflow state-machine basic transitions

Missing tests the audit found:

1. Builder pack resolution end-to-end
2. `executor.dry_run()` creates synthetic outputs with
   `dry_run=True` flagged
3. Illegal state transitions are rejected
   (you cannot `APPROVED → DRAFT`)
4. Rollback plan inverts commands correctly per pack
5. `request_approval()` fires the configured notifier
6. Builder AI fallback called when packs miss
