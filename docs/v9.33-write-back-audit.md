# v9.33 #1 — Identity write-back audit

This is the trust-posture inventory before v9.33 lands. Every path
that mutates an external identity system was inspected for default
behavior, gating, and operator visibility.

## Surface map (all five identity targets)

| Layer | Symbol | Default |
|-------|--------|---------|
| Adapter | `IdentityWriteBackMixin.apply_policy(ir, *, dry_run=True)` | dry-run |
| Transactional | `identity.transactional.apply_all(ir, adapters, dry_run=True)` | dry-run |
| HTTP | `POST /api/identity/apply` — `dry_run = body.get("dry_run", True)` | dry-run |
| HTTP | `POST /api/identity/apply-all` — `body.get("dry_run", True)` | dry-run |
| HTTP | `POST /api/identity/auto-fix/{id}` — `dry_run: bool = True` | dry-run |
| CLI | `safecadence identity apply` — `--dry-run/--commit, default=True` | dry-run |
| UI | `/identity` auto-fix fetches `?dry_run=true` explicitly | dry-run |

JIT operations are **decoupled from apply** by design:

| Endpoint | Effect |
|----------|--------|
| `POST /api/identity/jit/grant` | Persists a `JITGrant` record only |
| `POST /api/identity/jit/revoke/{grant_id}` | Marks grant `revoked` only |
| `POST /api/identity/jit/expire-due` | Daemon-driven status update only |

To actually commit a JIT operation to a target system, the operator
must call `/api/identity/apply` with the resulting IR.

Tier-3 SSH execution (network-side) is gated independently and is
already enterprise-grade: APPROVED job + TOTP MFA + `acknowledge` +
`i_mean_it` triple-flag. Not in scope for v9.33.

## Findings

1. **Defaults are correct everywhere.** No code change is required
   to keep dry-run the default. Every layer already honors it.

2. **Operator has no visible "what would change" surface.** The
   dry-run response carries `pending_ops` but the UI doesn't render
   a per-system diff card before the operator clicks Commit. This
   is the highest-leverage trust gap — it's where unknown software
   loses the buyer's confidence.

3. **JIT grant → apply chain is implicit.** A user who clicks
   "approve grant" gets a persisted record but no automatic dry-run
   against the target system. They have to know to call
   `/api/identity/apply` with the JIT IR. UI should chain
   automatically: grant → dry-run → diff → confirm → apply.

4. **Apply doesn't require proof the dry-run was viewed.** Anyone
   with a writer token can POST `apply: true` straight at
   `/api/identity/apply`, skipping the dry-run review entirely.
   This is the trust hole #2 closes.

5. ~~**Transactional rollback covers all 5 write-back targets** via
   `apply_all()` + the mixin's `_rollback()` hook. Confirmed.~~

   **Correction (during v9.33 #3):** the audit was wrong here. None
   of the 5 real adapters (Okta, ISE, AD/LDAP, Entra, ClearPass)
   actually implemented `_rollback()`. The transactional layer
   silently logged "adapter has no _rollback() hook" on partial
   failure and left committed work in place. Closed in v9.33 #3:
   each adapter now ships a real `_rollback()` that DELETEs (or
   MODIFY_DELETEs for AD) the IDs it previously committed; tests
   pin the URL/DN shape per system, including a transactional
   end-to-end test that proves the real Okta path runs.

## v9.33 #2 plan (consequence of this audit)

Instead of just "flip default to dry-run" (already done), close
the proof-of-review hole:

- Every dry-run response returns a `confirm_token` =
  HMAC(secret, ir_hash + adapter_version + actor + ts).
- Every apply requires `confirm_token` in the body. Commit fails
  with 409 if missing, mismatched, or older than 10 minutes.
- Apply audit log records both the dry-run hash and the apply
  hash so any inconsistency is detectable post-facto.
- CLI carries the same gate via `safecadence identity apply` —
  the command runs the dry-run automatically, prints the diff,
  and prompts for confirmation before committing.

## v9.33 #3 plan

- Reusable diff-card component on `/identity` rendering
  `pending_ops` per system with a colored severity pill,
  collapsible per-op detail, and a single "I've reviewed this —
  commit" button that posts the `confirm_token`.
- Audit transactional `_rollback()` for AD/LDAP, ISE, Entra,
  ClearPass; ensure each target produces a rollback receipt
  surfaced on the diff card after a partial-failure rollback.

Trust property of the resulting flow: **no external mutation can
happen without (a) a dry-run record and (b) an operator clicking
through the per-system diff.** This is the gold standard for
unknown security software touching production identity.
